# Async-mode FSP on Kuhn Poker — operational notes

## How to run

Smoke (5 steps, batch 16, 2 cold-starts):

```bash
sbatch scripts/auton/run_kuhn_poker_async_smoke.sh
# → examples/agentic_demo/agent_kuhn_poker_fsp_async_smoke.yaml
```

Full (300 steps, batch 128): copy `run_kuhn_poker_full.sh`, point `--config_name` at a new async yaml derived from `agent_kuhn_poker_fsp_train.yaml` that adds `async_generation_ratio: 1`.

## The only config knob

`async_generation_ratio: 1` (int, ≥0). `1` = vLLM generates one step ahead of the trainer. Speedup bounded by `sync_step / max(rollout_time, trainer_serial_time)` — on the smoke config that's ~1.25×, not 2×. Trainer serial path (ref/old_log_probs + train) is ~33 s and dominates; rollout (~9 s) is fully hidden but small. Bigger speedup requires rebalancing (longer sequences, smaller trainer, bigger effective train batch), not a larger ratio.

## FSP cold-start — required code path (already landed)

FSP resets trainer LoRA to zero every `fsp_save_steps`. In async mode, rollouts generated by the pre-reset model can survive in the queue into the post-reset training batch — the PPO ratio would diverge and training diverges on data its current weights never produced.

Fix implemented (`agentic_pipeline.py:604-610`, `rollout_scheduler.py`):

1. At cold-start: `reset_lora_weights` + set `self._pending_fsp_flush = True`. Don't suspend or flush here — doing so mid-step races vLLM's abort cleanup and deadlocks step N+1's `model_update`.
2. At step N+1, **after** `model_update` has pushed the reset LoRA to vLLM and **before** `get_batch`: check the flag and call `train/val_rollout_scheduler.flush_pending.remote()`. vLLM is idle-with-new-weights at that moment, so the drain races nothing.
3. `flush_pending` uses `GroupQueueManager.flush_in_place` (not `clear`) — `clear` replaces the `progress`/`complete` asyncio.Events and strands any es_manager worker already inside `await event.wait()` on the old Event. `flush_in_place` keeps Event identity and `.set()`s them so waiters re-loop.

All three steps are guarded by `if self.pipeline_config.async_pipeline:` — sync mode skips the whole path.

Log signature when healthy:

```
[agentic_pipeline.py (605)] FSP cold_start: resetting training LoRA weights at step N
[agentic_pipeline.py (682)] pipeline step N finished
[agentic_pipeline.py (327)] FSP force-sync: flushing pre-reset rollouts at step N+1
[agentic_pipeline.py (682)] pipeline step N+1 finished   # must appear within ~60s
```

If step N+1 logs the flush but never prints `pipeline step N+1 finished`, the Event-identity fix regressed — check `flush_in_place` vs `clear` in `RolloutScheduler.flush_pending`.

## Hard constraints

- **Device mapping**: disjoint train + infer (Configuration Model A) works with any `async_generation_ratio ≥ 0`. `partial_gpu_mode` (train ⊂ infer) **requires** async > 0 and ≥2 infer DP ranks (`agentic_pipeline.py:937-940`). Colocated mode (train == infer) **forbids** async (line 996).
- **Never run on rhea head.** Always `sbatch`.
- **Scratch only** for outputs — home is ~85% full.
- **CUDA env block** (`CUDA_HOME`, `CPATH`, `LIBRARY_PATH`, `TRITON_CACHE_DIR`, `RAY_TMPDIR`) — already baked into sbatch template; don't strip.
- **wandb project**: `self-play` for real runs, `self-play-debug` for smokes. `exp_name` must change across hyperparam-different runs.

## What to watch (wandb)

Async stays on-policy enough that PPO ratio should still hover at 1. Flag any row where:

| metric | anomaly |
|---|---|
| `actor/ratio_mean@mean` | drift outside 0.95–1.05 |
| `actor/ratio_max@max` | > 2 |
| `actor/approxkl@sum` | > 0.05 |
| `actor/ppo_ratio_clipfrac@sum` | > 0.3 |
| `actor/pg_loss@sum` | abs > 5 |
| `actor/entropy@sum` | collapse < 0.3 |

**Extra check**: the step **right after each force-sync flush** (N+1 after every cold-start). If the flush is working, that step's ratio_mean is ~1.0 and clipfrac is low. If flush regressed, expect ratio_max >> 1 and clipfrac spike at exactly those steps.

## Timing cues

- `time/step_rollout` ~0.5 s = async working (pre-gen'd batch pulled from queue).
- `time/step_rollout` ~9 s = cold step (startup or right after a force-sync flush).
- First step after each cold-start is always "cold" — that's the force-sync tax. At `fsp_save_steps=50` over 300 steps: ~6 cold steps, negligible overall.

## Known deadlocks (fixed, for postmortem reference)

- **`ray.get(suspend)` in force-sync path** (initial attempt): deadlocked step N+1's `model_update` because vLLM abort cleanup was still in progress when NCCL broadcast opened. Fix: move flush to step N+1 after model_update, drop the extra suspend.
- **`GroupQueueManager.clear()`** replaces asyncio.Events: waiters on the old Event never wake; get_batch hangs forever. Fix: `flush_in_place` preserves Event identity.

If a future change needs a similar "drain queue mid-step" primitive, reuse `flush_in_place`, not `clear`.
